# Decision Guide: Data Provenance and Input Identity Verification

## The Decision

How should a pipeline verify the identity and provenance of its input data to prevent silent substitution or corruption?

## When This Applies

- Pipelines that consume external data files (CSVs, database dumps, API snapshots)
- Systems where input files are manually placed or transferred
- Projects where input data changes between versions and version matters
- Pipelines that have experienced unexpected output changes due to input substitution

## Options

### Option A: Trust File Path
Use the file path as the sole identifier. If the file is at the expected path, trust its contents.

- **When to choose**: Inputs are generated by automated, reliable pipelines with no manual intervention
- **Pros**: Simple. No additional verification steps.
- **Cons**: No protection against substitution, corruption, or version mismatch. Silent failures.

### Option B: Manifest with Checksums
Maintain a manifest file listing each input with its expected checksum (SHA-256), row count, and column headers.

- **When to choose**: Inputs change occasionally, need verification without full data versioning
- **Pros**: Detects substitution and corruption. Low implementation effort. Audit trail.
- **Cons**: Manifest must be updated when inputs legitimately change. Checksum computation adds pipeline latency.

### Option C: Content-Addressed Storage
Store inputs in a content-addressed system (e.g., DVC, Git LFS, or a custom hash-indexed store). Reference inputs by content hash, not path.

- **When to choose**: Inputs change frequently, need full version history, team has infrastructure capacity
- **Pros**: Immutable inputs. Full version history. No accidental substitution possible.
- **Cons**: Higher infrastructure complexity. Storage overhead for large files. Team must adopt new tooling.

## Recommendation

Default to **Option B** — a manifest with checksums provides strong protection with minimal tooling. Implement verification as a pre-processing step that aborts the pipeline on mismatch.

Upgrade to **Option C** when: (1) inputs are large (>1GB), (2) multiple versions must be maintained simultaneously, or (3) reproducibility across long time horizons is required.
