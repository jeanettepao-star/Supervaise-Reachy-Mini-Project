# Test Pattern: Ground-Truth Comparison

## Purpose

Validate pipeline output correctness by comparing against an independently verified baseline (ground truth). This catches systematic errors that per-row validation misses — wrong transformations applied consistently, entire categories missing, or values shifted by a constant offset.

## When to Use

- After building or modifying a data transformation pipeline
- When output correctness cannot be verified by schema validation alone
- When the pipeline applies non-trivial transformations (aggregation, normalization, filtering)
- As a regression baseline for future pipeline changes

## Methodology

### Step 1: Establish Ground Truth

Create a small, manually-verified baseline dataset:
1. Select a representative subset of input data (3-5 entities across different categories)
2. Manually compute the expected output for each entity using the authoritative calculation method
3. Document the calculation steps and authoritative source
4. Store as a versioned fixture file (e.g., `tests/fixtures/ground-truth-{name}.csv`)

### Step 2: Automate Comparison

Write a test that:
1. Runs the pipeline on the same input subset
2. Loads the ground truth fixture
3. Joins pipeline output with ground truth on entity key
4. Asserts equality (or near-equality for floating point) on every output column
5. Reports specific discrepancies with context (entity, column, expected, actual)

### Step 3: Maintain Baseline

- When the pipeline's business logic changes, update the ground truth fixture
- When new categories or entities are added, extend the fixture
- Never auto-generate ground truth from the pipeline itself — that defeats the purpose

## Key Principles

- **Independent verification**: Ground truth must be computed independently of the pipeline code. Manual calculation or a separate trusted tool.
- **Representative coverage**: Include edge cases — entities with missing values, extreme values, boundary conditions.
- **Specific errors**: Report which entity and which column disagrees, not just "output doesn't match."
- **Version discipline**: Ground truth fixtures are versioned alongside the pipeline. Changes to fixtures require justification.

## Anti-Pattern: Self-Referential Baseline

Creating ground truth by running the pipeline and saving its output as the baseline. This validates consistency (same output as before) but not correctness (output matches reality).

**Detection**: Look for test fixtures that were generated by the pipeline itself. Check fixture commit history — if fixtures and pipeline code changed in the same commit with no external reference, suspect self-referential baseline.
