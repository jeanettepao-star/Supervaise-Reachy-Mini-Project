# Anti-Pattern: Hand-Edited Generated Artifact

## The Pattern

A file is produced by a generator script (from a template, frontmatter, or source data) but reviewers edit it directly to fix small issues — typos, formatting, row reordering. The generator's next run overwrites the fixes. The cycle repeats.

## Symptoms

- Commits that edit generated files get silently reverted on the next generator run
- Fixes reappear in bug reports a week after being "fixed"
- Frustration with "the tool keeps undoing my work"
- Reviewers cannot tell which files are generated and which are authored
- Hand-edits fix symptoms; the underlying generator bug is never addressed

## Root Cause

Generator outputs have no single-writer guard; hand-edits are structurally possible because nothing prevents them. Without a header declaring provenance, even conscientious reviewers cannot tell which files they are allowed to edit.

## Prevention Rules

1. **Hash header on every generated file** — A machine-readable comment declaring generator, source hash, and timestamp. Example:
   `<!-- generated: {ISO8601} | source-sha: {sha256} | generator: {script}@v1 -->`
2. **Pre-commit hook rejects stale headers** — Recompute the source hash; if it doesn't match the header, the file is either stale or hand-edited. Commit fails with a clear message directing the user to regenerate.
3. **Fix the source, not the output** — If a fix is needed, change the generator's input (frontmatter, template, source data) and regenerate. Never edit the output directly.
4. **Visually distinctive generated files** — The hash header at the top is an obvious visual signal: this file is not for hand-editing.

## Detection

- Pre-commit hook + hash header verification (primary)
- Reviewer inspection: does the file carry a "generated by" header? If yes, do not edit.
- CI check: recompute every generated file; any diff = drift = fail
